# Planner Bead Sizing and Coalesced Followups

## Objective

Change how the planner decomposes specs into beads so that developer beads are small and focused (under 10 minutes each), and test/review/docs beads are coalesced per feature rather than spawned per developer bead.

## Why This Matters

Currently, the planner creates large developer beads scoped by logical boundary (e.g. "UUID storage layer" or "wire config into runners and CLI"). These often take 10-15 minutes and risk timing out. Each developer bead also auto-generates 3 followup children (test, docs, review), leading to many small review/docs beads that each see only a fragment of the feature.

This is wasteful:
- Developer beads are too large, causing timeouts and looping
- Review beads review tiny incremental changes without full feature context
- Documentation beads document fragments rather than the complete feature
- A feature with 5 developer beads generates 15 followup beads (5x3), most of which are trivially small

## Proposed Model

### Small developer beads

The planner should decompose implementation work into beads that each:
- Change no more than 2-3 functions or a single focused concern
- Are implementable in under 10 minutes by an agent
- Have clear, narrow acceptance criteria

For example, instead of one bead "UUID ID generation and prefix resolution in storage layer", the planner would create:
- "Replace sequential ID allocation with UUID generation" (one function)
- "Add prefix matching for partial bead IDs in storage" (one function)
- "Update child bead ID derivation for UUID format" (one function)

### Coalesced followup beads

Instead of the scheduler auto-creating test/docs/review children for each developer bead, the planner should explicitly create shared followup beads that depend on all related developer beads:

```
Feature Root
├── Dev bead 1 (small, focused)
├── Dev bead 2 (depends on 1)
├── Dev bead 3 (independent of 2)
├── Test bead (depends on 1, 2, 3)
├── Review bead (depends on 1, 2, 3)
└── Docs bead (depends on 1, 2, 3)
```

Benefits:
- The tester sees all implementation changes together and can write comprehensive tests
- The reviewer sees the complete feature and can assess architectural coherence
- The docs bead documents the feature holistically, not piecemeal
- Fewer total beads (N devs + 3 followups instead of N devs + 3N followups)

### When to still use per-bead followups

The planner should use coalesced followups for most features. Per-bead followups (auto-generated by the scheduler) remain appropriate when:
- A developer bead is standalone with no related beads in the same feature
- The bead is a hotfix or corrective that needs immediate isolated validation

## Functional Requirements

### 1. Planner Template Update (`templates/agents/planner.md`)

Add sizing and followup guidance:

- Each developer bead should be scoped to a single focused change, implementable in under 10 minutes
- If a logical unit requires changes to more than 2-3 functions or touches multiple subsystems, split it into smaller beads with dependencies
- Create shared test, review, and documentation beads that depend on all related developer beads in the feature, rather than relying on per-bead auto-followups
- Test beads should depend on all developer beads they validate
- Review and documentation beads should depend on all developer beads plus the test bead

### 2. Planner Prompt Update (`prompts.py`)

Update `build_planner_prompt()` to reinforce the sizing constraint. The prompt already asks for bead structure — add guidance that implementation beads should be small and followup beads should be coalesced.

### 3. Scheduler Followup Suppression

The scheduler currently auto-creates test/docs/review children whenever a developer bead completes (`_create_followups()` in `scheduler.py`). This should be suppressed when the planner already created explicit followup beads in the feature tree.

Detection: if the feature root already contains test/review/docs beads that list the completing developer bead in their dependencies, skip auto-creation. This prevents duplicate followup beads.

### 4. Backward Compatibility

- Beads created before this change (with auto-followups) continue to work
- Manually created developer beads (via `bead create`) with no planner-provided followups still get auto-followups from the scheduler
- The planner can still create per-bead followups when appropriate (standalone beads, hotfixes)

## Acceptance Criteria

- Planner produces developer beads scoped to single focused changes
- Planner creates coalesced test/review/docs beads per feature, not per developer bead
- Coalesced followup beads depend on all related developer beads
- Scheduler does not auto-create followup children when planner-provided followups exist
- Standalone developer beads without planner followups still get auto-followups
- All existing tests pass
