# Investigator Bead Type for Open-Ended Codebase Analysis

## Objective

Operators need a way to run open-ended analytical tasks — technical debt surveys, dependency audits, coverage analysis, API surface reviews — without routing them through a developer or review bead. The existing types all carry semantics that break down for read-only investigation: `developer` triggers test/docs/review followups, `review` requires a verdict, `planner` produces a bead graph. A dedicated `investigator` type provides a clean home for these tasks: read freely, write one report file, produce structured findings, stop.

## Problems to Fix

### 1. No suitable bead type for read-only analysis

Operators currently have no correct type for investigation work. Using `developer` creates unwanted followup beads; using `review` forces a verdict and scopes the agent to a specific changeset; using `documentation` is semantically wrong and pushes the agent to write docs rather than findings.

### 2. No structured findings format

Ad-hoc analysis beads (however created) have no standardised output schema. Findings, recommendations, and risk areas are buried in free-text handoff summaries that aren't queryable or reusable.

### 3. No persistent report artifact

Investigation results live only in bead metadata and are lost to context once a conversation ends. There is no durable, human-readable report that an operator can read, share, or use as input to a spec.

## Changes

### 1. New agent type: `investigator`

**File: `src/agent_takt/models.py`**

Add `"investigator"` to `AGENT_TYPES`. Do NOT add it to `MUTATING_AGENTS` — investigator beads are read-only.

### 2. Output schema for investigator beads

**File: `src/agent_takt/runner.py`** — `AGENT_OUTPUT_SCHEMA`

Add `"investigator"` to the schema's `agent_type` enum. The investigator output schema differs from the standard worker schema: it omits `changed_files`, `verdict`, and `next_agent`, and adds investigation-specific fields:

```json
{
  "outcome": "completed | blocked",
  "summary": "...",
  "findings": "...",
  "recommendations": "...",
  "risk_areas": "...",
  "report_path": "docs/investigator/<slug>.md"
}
```

`findings`, `recommendations`, and `risk_areas` are required strings. `report_path` is the path to the written report file relative to the repo root.

### 3. Guardrail template

**File: `templates/agents/investigator.md`**

New guardrail template. Key constraints:
- Read freely: `Read`, `Glob`, `Grep`, `Bash` (read-only commands — `git log`, `wc`, counting, searching). No `Write` or `Edit` except to the single declared report file.
- The report file path is provided in the bead description or as `expected_files[0]`. Write the report there and nowhere else.
- Report structure: `## Findings`, `## Recommendations`, `## Risk Areas` sections, with a one-paragraph executive summary at the top.
- Do not create sub-beads, do not modify source files, do not run the test suite.
- The handoff must populate `findings`, `recommendations`, `risk_areas`, and `report_path`.

### 4. Report file output

**Convention:** report files live at `docs/investigator/<slug>.md` where `<slug>` is derived from the bead title (lowercase, hyphens). The report path is declared in `expected_files` on the bead so the scheduler treats it as the bead's write scope. This prevents two concurrent investigations from writing to the same file while leaving the read scope unconstrained.

The report file format:

```markdown
# <bead title>

*Generated by investigator bead <bead_id> on <date>*

## Executive Summary

<one paragraph>

## Findings

<detailed findings>

## Recommendations

<prioritised list>

## Risk Areas

<what is risky if left unaddressed>
```

### 5. Scheduler: no followup creation for investigator beads

**File: `src/agent_takt/scheduler/followups.py`**

Investigator beads must not trigger `-test`, `-docs`, `-review` followup creation. Add `"investigator"` to the existing exclusion check alongside `"review"` (which also skips followup creation).

### 6. Skill allowlist

**File: `src/agent_takt/skills.py`**

Add an `investigator` entry to `AGENT_SKILL_ALLOWLIST`. Allowlist: `core/base-orchestrator`, the investigator role skill, and `memory`. No capability or task skills that imply code mutation.

**File: `.agents/skills/role/investigator.md`** (new)

Role skill describing the investigator's mandate: read widely, synthesise findings, write one report, stop.

### 7. CLI usage

No CLI changes required — `--agent` already accepts any string from `AGENT_TYPES`. Adding `investigator` to `AGENT_TYPES` is sufficient. Example:

```bash
uv run takt bead create \
  --agent investigator \
  --title "Technical debt survey: scheduler package" \
  --description "Survey src/agent_takt/scheduler/ for technical debt: duplication, complexity hotspots, missing abstractions, and test coverage gaps. Write findings to docs/investigator/scheduler-technical-debt.md" \
  --expected-files docs/investigator/scheduler-technical-debt.md
```

### 8. Config: add `investigator` to `agent_types`

**File: `.takt/config.yaml`**

Add `investigator` to the `common.agent_types` list to keep `test_repo_config_matches_defaults` passing (same pattern as `recovery`).

## Acceptance Criteria

- `takt bead create --agent investigator` creates a bead without error
- When an investigator bead completes, no `-test`, `-docs`, or `-review` followup beads are created
- The investigator output schema requires `findings`, `recommendations`, `risk_areas`, and `report_path`; `verdict` and `changed_files` are absent
- The guardrail template prevents writing to any file other than the declared report file
- `docs/investigator/` is the canonical report directory; a completed bead writes its report there
- `test_agent_types` and `test_repo_config_matches_defaults` pass with `investigator` included
- An investigator bead with overlapping `expected_files` is blocked by the scheduler's conflict check in the normal way; an investigator with no `expected_files` is never blocked on file scope

## Pending Decisions

### 1. Output schema: single schema or variant?
The current `AGENT_OUTPUT_SCHEMA` is shared across agent types with an `agent_type` enum. Investigator output differs enough (no `verdict`, no `changed_files`, adds `findings`/`recommendations`/`risk_areas`) that it may need a separate schema object or a conditional branch. **Leans toward a separate schema keyed by agent type, applied at dispatch time — same pattern as planner.**

### 2. `expected_files` requirement
Should the CLI enforce that investigator beads declare exactly one entry in `expected_files` (the report path)? Enforcement would prevent beads with unconstrained write scope but adds complexity. **Leans toward convention over enforcement — the guardrail and docs make the expectation clear; no CLI validation added initially.**

### 3. Role skill scope
The investigator role skill could be minimal (just mandate + output format) or include investigation patterns (how to structure a codebase survey, what to look for in a dependency audit, etc.). **Leans toward minimal — keep the guardrail prescriptive and the role skill brief.**
